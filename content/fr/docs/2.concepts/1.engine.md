## Vue d’ensemble

**Worcable** est un **ATS (Applicant Tracking System) open source** basé sur une architecture modulaire, conçu pour être :

- **Auto-hébergeable**
- **Extensible via services tiers**
- **Orienté événements (event-driven)**
- **API-first**

Le système est composé de trois blocs principaux :

```
[ Client (UI) ]  →  [ API Worcable ]  →  [ Database ]
                         ↓
                 [ Services tiers ]
```

## Architecture globale

### 🔹 Core API (cœur du système)

L’API Worcable est le **point central** :

- Expose une **API REST**
- Contient toute la **logique métier**
- Gère **l’accès à la base de données (unique point d’accès)**
- Orchestre les interactions avec les services via des **événements**

::callout{icon="i-lucide-chevrons-right"}
**Principe clé :**
Aucun service externe n’accède directement à la base de données
::

### 🔹 Client (Frontend)

Le client est responsable de :

- L’interface utilisateur (gestion du recrutement)
- L’orchestration des actions côté utilisateur
- L’intégration et la configuration des services tiers
- La génération de **tokens d’accès API**

::callout{icon="i-lucide-chevrons-right"}
Le client est le **point d’entrée opérationnel** pour les utilisateurs et les intégrations.
::

### 🔹 Services tiers (extensibilité)

Les services tiers permettent d’étendre Worcable sans modifier le core :

Exemples :

- CV Parser (Python)
- Mailer
- Notifications
- IA / scoring
- Intégrations externes (HR, job boards…)

#### Caractéristiques :

- Indépendants (microservices)
- Connectés via **events**
- Enregistrés dynamiquement

## Fonctionnalités natives de l’API

L’API Worcable gère nativement :

| Catégorie                       | Fonctionnalités                                                                              |
| :------------------------------ | :------------------------------------------------------------------------------------------- |
| **Authentification & sécurité** | • Gestion des utilisateurs<br>• Authentification (JWT ou équivalent)<br>• Gestion des accès  |
| **Gestion des offres**          | • Création / édition d’offres<br>• Statuts (ouverte, fermée, etc.)                           |
| **Pipeline de recrutement**     | • Définition de workflows personnalisés par offre<br>• Étapes configurables (kanban logique) |
| **Gestion des candidatures**    | • Réception des candidatures<br>• Centralisation des profils<br>• Historisation              |
| **Évaluation**                  | • Notation des candidats<br>• Ajout de commentaires<br>• Collaboration interne               |

## Système d’extensibilité (Services tiers)

### Ajout d’un service

Le processus d’enregistrement est **sécurisé et découplé** :

1. Le client demande à l’API un **token d’initialisation**
2. Le client transmet ce token au service tiers
3. Le service appelle l’API avec :

   - Le token
   - Sa configuration

4. L’API enregistre le service

::callout{icon="i-lucide-check-check" color="success"}
**Cela garantit** :

- Un contrôle strict des services autorisés
- Une configuration dynamique
::

### Communication : Event-driven

L’API ne fait **aucun appel direct** aux services.

::callout{icon="i-lucide-chevrons-right" color="success"}
**Elle émet des événements :**

- Exemples :
  - `candidate.created`
  - `application.received`
  - `candidate.scored`
    ::
::

:icon{name="i-lucide-chevrons-right"} Les services :

- S’abonnent aux événements
- Réagissent de manière asynchrone

:icon{name="i-lucide-chevrons-right"} Avantages :

- Découplage total
- Scalabilité horizontale
- Résilience (pas de dépendance synchrone)
- Facilité d’ajout/suppression de services

## Gestion des accès API

Le client peut générer des **tokens d’accès API** permettant :

- D’interroger l’API Worcable
- D’intégrer des systèmes externes
- D’automatiser des workflows

:icon{name="i-lucide-chevrons-right"} Modèle sécurisé basé sur :

- Tokens scoped (recommandé)
- Permissions contrôlées

## Gestion des données

### Base de données

- Accès **exclusif via l’API**
- Aucune connexion directe externe
- Centralisation des règles métier

:icon{name="i-lucide-chevrons-right"} Garantit :

- Cohérence des données
- Sécurité
- Auditabilité

## Services natifs

Certains services sont **préinstallés par défaut** :

- CV Parser
- Mailer

::callout{icon="i-lucide-lightbulb" color="info"}
Ces services sont considérés comme **first-class citizens**, mais restent techniquement découplés.
::

## Cas d’usage typique

Exemple de flow :

1. Un candidat postule
2. L’API enregistre la candidature
3. L’API émet `application.received`
4. Le CV Parser :

   - récupère le CV
   - extrait les données

5. Un service IA :

   - score le candidat

6. Un service mail :

   - envoie un accusé de réception

::callout{icon="i-lucide-chevrons-right" }
Aucun couplage direct entre services
::
