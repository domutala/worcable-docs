Worcable’s architecture is designed to be **modular, scalable, and monorepo-oriented**, to facilitate the development, deployment, and maintenance of each component.

It is based on a clear separation between the application core, specialized services, and the deployment infrastructure.

## Overall Architecture

Worcable is organized as a **structured monorepo**, enabling unified source code management:


```text [Structure]
.
├── package/      # Main application (Nuxt 3 full-stack)
├── cvparser/     # Standalone CV parsing service (Python)
├── deploy/       # Infrastructure and Docker orchestrator
└── docs/         # Technical documentation (Nuxt Content)
```

## Monorepo Applications

Worcable separates the application core from **Independent Services**, allowing specific processes to be isolated.

| Module        | Role | Details | Purpose |
|--------------|------|--------|----------|
| `package/` | Core Application | - UI (pages, components)<br>- Backend API (`/server/api`)<br>- Business logic<br>- Database access | Core of the product |
| `docs/` | Documentation | - Based on **Nuxt Content**<br>- Versioned documentation<br>- i18n support | Clean and scalable DX |
| `cvparser/` | Parsing Service | - PDF / DOCX parsing<br>- Data extraction<br>- Candidate profile structuring<br><br>**Stack:**<br> - Python<br>- FastAPI<br>- NLP (spaCy, etc.) | Performance, scalability, AI evolution |
| `deploy/` | Infrastructure | - `docker-compose.yml`<br>- Dockerfiles (base + services)<br>- Launch scripts | Reproducible deployment |


## Scalability

Worcable is built on a **modular monolithic** architecture. The frontend and backend coexist in `package/` while maintaining a strict separation of responsibilities. This core delegates resource-intensive tasks to **specialized services** (such as `cvparser`), allowing heavy computations to be offloaded.

The whole system is orchestrated via **Docker**, ensuring consistent deployment across any environment, from VPS to the cloud. This structure allows for evolution toward a distributed system: it is possible to introduce message queues for asynchronous processing or to extract logical components into dedicated frameworks if the load demands it.

## Deployment

Worcable offers complete flexibility for production, adapting to your infrastructure constraints through a unified orchestrator:

### Option 1: Docker (Recommended)
Full orchestration via Docker Compose. This is the preferred method for isolation and easy scaling.
```bash [Terminal]
# Launch everything (Nuxt, DB, Parser)
./deploy/run.sh all

# Launch specific services (e.g., Nuxt + MongoDB)
./deploy/run.sh base db

# Launch and view logs for a specific service
./deploy/run.sh cvparser --logs
```

::callout{icon=“i-simple-icons-docker” color="info" to="/en/docs/deployment/docker"}
Read the complete guide to deploying Worcable with Docker in production environments.

### Option 2: Native (Shell Script)
For environments without Docker. Requires **Node.js (pnpm)**, **Python 3**, and **MongoDB** installed locally.
```bash [Terminal]
# Launch everything natively
./deploy/run.sh all --native

# Launch only the Nuxt application
./deploy/run.sh base --native

# Launch specific services natively
./deploy/run.sh base cvparser --native
```

## Philosophy

Worcable is designed to:

  * **Stay simple** for open-source contributors to understand.
  * **Be quick** to deploy without complex configuration.
  * **Ensure data sovereignty** through self-hosting.
  * **Adapt** to resource-constrained environments.
